[개요]

편입 첫학기에 들었던 자료구조 과목의 정리노트이다. 코딩실력이 한참 부족할 때지만, 나만의 작업물을 만들고 싶은 열망으로 시작했다. 각 주차별 코드는 깃허브에, 상세한 풀이과정은 네이버 블로그에 서술해 두었다.
링크URL: [Github] [Naver blog]

[개발내용]

개념을 익히는 문제들은 다음과같이 풀이과정이 남는다. 각 주차마다 1개에서 많게는 10개남짓의 문제가 과제로 출제되었다. 대부분 표의 빈칸에 문자를 채워넣거나, 자료구조를 그림으로 그려서 구현하는등의 문제였다. 나는 파워포인트를 사용해 풀이과정 이미지를 만들고 블로그에 글로써 서술해 풀이과정을 남겼다. 중요한 개념을 배울때마다 각 주차에는 항상 코딩과제가 있었다. 코딩과제는 대부분 교재의 코드에서 특정 부분의 기능을 향상하거나 또는 그 주차의 자료구조를 활용하여 간단한 동작을 하는 프로그램을 구현하는것이었다. 다음과같이 중간과정을 서술해야 하는 문제는 풀이가 매우 길어졌다(이하생략)

[느낀점]

수업이 점점 진행되면서 앞에서 배웠던 내용을 뒤에서도 써야할 때가 있었는데, 그 때마다 강의노트나 교과서를 다시 보지 않고 내가 사용했던 코드나 풀이를 보면서 많은 도움을 얻을 수 있었다. 나는 자료구조 수업을 듣기 전에 "생활코딩"에서 베포한 Web 강의를 듣고 html을 배웠었는데, 처음으로 블로그 포스팅을 하면서 웹 기반의 문서편집기가 가진 기능들이 정말 대단하고 신기한 기능이 많다고 느꼈다. 필요에 따라서 코드와 이미지, 글을 복합적으로 표현하고 스타일을 지정할수 있다는게 내가 처음으로 컴퓨터 공학 공부를 시작하면서 프로그램의 원리에 대해 생각해보고 감탄했던 순간이었다. 특히 Web 기반의 문서는 언제 어디서나 접근이 가능해서, 내가 자료구조 기말고사를 보러가는 동안에도 스마트폰으로 요약노트를 보며 공부할수 있다는게 혁신적이었다.

지금은 C언어를 사용한 프로그래밍을 자주 하지 않아 다른 문서들을 참고하지만, 각 자료구조들에 대한 개념과 원리를 이 떄 잘 정리해놓고 반복해서 보았기에 현재 조금이나마 적재적소에 알맞은 자료구조를 사용할 수 있게 되었다고 느낀다.

첫 작품(?)이기에 많이 부족하고 어디 내놓기 부끄럽지만 한편으론 뿌듯하다.

자동 ERP 매크로

구분 개인프로젝트(인턴)
개발기간 2개월
사용기술 Language: Python
Module: selenium, beautifulsoup, pyautoGUI
기타 산학 보안협약에 따라 private repository로 운용

[개요]

ERP 시스템의 발주 프로세스를 자동화하기 위한 매크로 프로그램이다.

[배경]

자사의 ERP 시스템의 중계자로서 고객이 주문을 요청하면 자사가 이를 확인하고 ERP프로그램, 웹사이트를 통해 배송요청을 진행해야했다. 기존엔 고객사와 납품사의 ERP시스템이 다르기 때문에 연동하는 부분이 인력으로 이루어지고 있었다. 따라서 매크로는 고객사 주문접수, 납품사 배송요청을 자동으로 연계하고자 설계되었다.

[개발내용]

매크로의 실행화면이다. Python이 설치된 환경의 컴퓨터에서 Cmd창을 통해 실행된다. 매크로의 동작은 크게 3가지로 구분된다. 차례대로 다음과 같다.

1. 오늘의 주문내역 가져오기(크롤링, txt파일에 쓰기)
고객사는 물품을 요청한 주문신청들을 받아서 관리하는 홈페이지가 있다. 이 홈페이지에서 selenium 모듈을 통해 하루치의 주문신청을 크롤링 한후 txt파일에 쓰도록 구성했다.

(고객사 A사의 홈페이지)
2. 주문내역에 따라 배송요청하기(크롤링, txt파일을 읽기)
이전 단계에서 저장해둔 txt파일을 읽어들여 납품사에 배송요청을 넣는부분이다.

웹페이지로 업무를 보는 X 납품사는 1단계와 동일하게 selenium 모듈을 사용했고, 페이지간 이동과 주문내용의 기입을 크롤링 기능으로 구현할 수 있었다.
(납품사 X사의 홈페이지)
ERP 프로그램으로 업무를 보는 Y 납품사는 기존의 웹 크롤링 기능을 사용할 수 없었다. 나는 실현가능한 대안으로 이미지 인식과 키보드 제어를 활용하는 pyautoGUI 모듈을 사용하기로 했고, 크롤러와 최대한 비슷하게 동작하도록 만들었다.
(납품사 Y사의 프로그램)
3. 배송요청 확인서와 주문내역을 대조하여 기록남기기(더블체킹)
최종적으로 프로젝트의 기능검토를 받고나서, 기록을 검토하여 정상동작을 확인할 수 있도록 검수기능을 추가하라는 피드백을 받았다.

아래는 해당 부분을 구현한 사진이다.
실제 주문서와 처리기록을 대조하여 수량,물품코드,이름이 모두 동일하면 <성공> 이라고 기록하게끔 구현했다.

[도전적 이슈&느낀점]

내가 사용가능한 프로그래밍 언어가 C와 Java밖에 없는 상황에서 Python언어를 처음으로 배워가며 시작했던 프로젝트이다. 업무로써 진행하는 첫 프로젝트인데 새로운 언어를 공부하며 매주 발전된 결과를 내야 한다는것이 조금 부담도 되었던 것 같다.

지금 돌이켜보면 실패의 부담이 없고 정말 쉬운 프로젝트였는데 괜한 부담을 가지고 힘을 쏟았구나 많이 느끼게 되는 프로젝트이다. Python에 입문하는데 큰 도움이 되었고, 매주 선임개발자 분에게 코드리뷰를 받으면서 개선할점을 들을 수 있어서 정말 좋은 경험이었다.

도로위험지수 기계학습 모듈제작

구분 팀프로젝트
내역할 -프로젝트 아이디어 제시(채택됨)
-도로위험지수 API 조사 및 데이터마이닝(공동)
-SVM 파라미터 튜닝(개인)
-SVM 사용 예측기능 구현(개인)
-클러스터링 기능구현(개인)
-발표자료 제작(공동)
개발기간 3개월
사용기술 Language: Python
Module: SVM, KNN, Sklearn, GridsearchCV
ETC: Google Colab

[개요]

도로교통 공단에서 제시하는 도로위험도 지수를 도출하는 모델을 기계학습을 통해 대체해보고 개선된 웹어플리케이션을 제시한다.

[배경]

한국교통연구원의 발표에 따르면, 한국의 2017년 교통사고 피해액은 40조 574억원에 달했다. 도로교통공단에서는 이러한 상황에 맞추어, ‘도로위험도지수’ 라는 지표를 산출하여 ‘위험도로 예보시스템’ 서비스를 운영하고 있다. 우리는 해당 도로위험도 지수 API를 가지고 차례로 3가지 목표를 세웠다.
1. 도로위험도지수에 대한 이해

2. 도로위험도지수 산출절차를 기계학습으로 대체할 모델 제시

3. 우리의 새로운 모델을 기반으로, 2021년 도로위험도 지도 API 제작

[개발내용]

기존 산출절차 방정식과 변수들을 정리해보았다.
도로위험도지수를 구하기 위해선 도로위험도 Basic을 먼저 산출해야한다. 도로위험도 Basic의 산출 방정식은 사진에 보이는 사고건수 사상자수 사상자환산계수를 통해 산출이되며 날씨,교통상황,시간은 고려되지 않는다.


수집한 data를 기반으로 도로위험도 basic에 따른 Feature값만 추출하였다. 선정한 feature에서 결측값은 존재하지 않는것으로 확인되었다.


다음으로 데이터의 값이 혹여나 너무 크거나 혹은 작은경우를 고려하여 데이터 스케일링을 진행했다. 데이터 스케일링에 대한 모델은 standard scaler, Robust Scaler, MinMax Scaler, Normalizer 총 4가지 모델이 고려되었으며 각각의 스케일 조정 전 후 최소,최대값을 비교해본 결과 차이가 가장 적은 RobustScaler로 선정하게 되었다.


알고리즘 선정을 위해선 KNN,SVM,DecisionTree,LinearRegression 알고리즘을 후보로 두고 학습 결과를 비교했다. 각각의 알고리즘에서 Best Hyper parameter를 찾기위해서 여러 차례 학습을 한 결과 가장 성능이 좋은 SVM으로 선정했다.


SVM의 정확도를 증가시키기 위해서 재차 튜닝할 필요가 있었고 best kernel을 다시 선정해야 했다. 3가지 커널을 비교하여 polynomial 커널이 가장 적합함을 판단했다.


어플리케이션의 구현에 앞서서 우리는 도로교통 공단이 제공하는 것 처럼 특정 위치의 도로위험도를 반환할 수 있는 API(모듈?)을 만들어여 했다. 위치별 도로위험도를 나타내기 위해서 위도경도정보를 사용하여 반경 1km 이내의 사고정보를 종합해야 했다. 그래서 클러스터링을 사용하는 방법으로 구현을 시도했다.


우리는 최종적으로 무작위의 300개 도로구간에 대한 도로위험도를 예측하도록 테스팅을 진행했다. 다음 값은 예측데이터와 타겟값으로 나누어 박스플롯으로 나타내었다. 정확도 0.85 이상의 목표치를 충족함과 동시에 전체적인 추이도 올바르게 진행되었으므로, 학습시킨 모델이 기존의 도로위험도 산출절차를 대체할 수 있다는 결론을 얻을 수 있었다.

[도전적이슈&느낀점]

우리팀은 목표했던 모델을 완성하였으므로 부가 과제인 개정판 도로위험도 지도를 만드는 단계를 시도해보았다.

기존의 K-Means 모듈을 통한 클러스터링은 전국지도를 격자로 나누어 클러스터링하려는 목적에 부합하지 않다고 판단되었다.

나는 위도경도를 통해 격자중심과 사고지점간의 km단위의 거리를 측정한후 반복문을 돌면서 1km이하의 거리를 판별해내는 코드를 제작하였다. 하지만 이 부분에서 딜레마에 빠지게 되었다.

전체 격자사이즈를 올리면 중심과의 거리차이는 줄어드나 러닝타임이 비정상적이게 길어지게 되고, 격자사이즈를 줄이면 러닝타임은 줄어들지만 거리차이가 벌어져 클러스터링에 유효한 군집화를 할 수 없었다.

이를 해결하고자 200*200 부터 600*600의 격자사이즈에 전국의 데이터 대신 서울시 중구, 노원구, 동대문구등 작은 지역의 데이터를 대입하여 러닝타임과 클러스터링간의 합의점을 찾고자 하였으나, 시간 부족으로 완성하지 못했다.

처음에 이 프로젝트의 기획안을 제시했을 때, 최종적으로 카카오 지도 API를 활용해 웹사이트까지 만드는것이 목표였는데 결국 필수항목들만 다 마친채로 프로젝트를 종료할 수 밖에 없어서 너무 아쉬웠다. 분명 이부분만 우리가 구현했었어도, 전체중에서 가장 참신하고 완성도 높은 프로젝트일 수 있었다고 생각한다.

앞으로는 완성된 샘플 프로젝트를 가진 상태에서 내 안건을 진행시켜야 겠다고 생각했다. 상당히 자신이 있던 프로젝트였는데, 두각을 드러내지 못한것이 뼈저리게 아쉬웠고 큰 교훈이 되었다.
